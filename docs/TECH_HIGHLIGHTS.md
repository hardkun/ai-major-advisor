# 技术亮点拆解

## 1. 后端架构

- 使用 FastAPI 构建 RESTful API。
- 按 routers / schemas / crud / services 分层组织代码。
- schemas 负责请求与响应结构，crud 负责 SQLite 数据访问，services 负责业务逻辑。
- 推荐、报告、采集、核验、数据源管理、覆盖率报告等模块拆分清晰。
- 接口中对不存在记录、AI 调用失败、数据源检测失败等情况做了错误处理和兜底。

## 2. 数据模型

- `schools`：学校基础信息。
- `majors`：专业基础信息和方向标签。
- `admissions`：正式推荐数据，只保存核验后的录取记录。
- `raw_admission_records`：原始采集数据，默认 pending，人工核验后才进入 admissions。
- `raw_data_sources`：数据源配置，包括 URL、解析器、字段映射、检测结果和候选状态。
- `collector_runs`：记录每次采集运行状态、插入数、跳过数和错误数。
- `reports`：保存免费报告和完整报告 JSON。

## 3. 数据采集

- 支持数据源搜索和候选源管理。
- 支持 URL 可采性检测，判断 HTML、CSV、Excel、PDF、附件页面等类型。
- HTML 采集器支持 table_index、表头自动识别、字段映射、合并单元格解析和宽表转换。
- Excel 解析使用 openpyxl。
- PDF 解析使用 pdfplumber。
- 附件型数据源可先提取文件链接，再生成子数据源处理。
- 采集预览接口不会写数据库，便于正式采集前检查字段映射和跳过原因。

## 4. 数据治理

- 自动采集数据进入 raw_admission_records，不直接进入推荐库。
- 管理员在 admin 后台查看详情、核验通过或驳回。
- 核验通过后写入 admissions，并标记 is_verified。
- 覆盖率报告用于查看学校数据覆盖情况。
- 缺口诊断用于区分未发现数据源、有源无数据、检测失败、解析失败等问题。
- 示例源、测试源和第三方候选源会被标记或排除，避免污染演示结果。

## 5. 推荐系统

- 输入字段包括省份、分数、位次、科类和目标方向。
- 支持“理科/文科”和“物理类/历史类”兼容。
- 根据专业方向标签过滤 AI 相关专业。
- 根据用户位次和往年最低位次判断冲 / 稳 / 保。
- 排序时优先考虑匹配类型，再考虑位次接近程度。
- 推荐结果包含最低分、最低位次、年份、专业代码、选科要求、校区、来源名称和核验状态。

## 6. AI 能力

- LLM 只负责解释推荐结果，不参与推荐排序。
- 生成字段包括推荐理由、学习重点、适合人群、就业方向和风险提示。
- 使用结构化 JSON 输出，方便前端和报告页展示。
- 如果 LLM 调用失败或 JSON 解析失败，会返回默认解释，不影响 recommend 主流程。
- 所有 AI 输出都包含“不构成最终志愿填报建议”的风险提示。

## 7. 前端与后台

- 微信小程序包含首页输入页、推荐结果页、报告详情页、合规说明页。
- 小程序支持 use_ai 开关、无匹配结果提示和 mock-pay 完整报告解锁流程。
- admin 后台使用原生 HTML / CSS / JavaScript，不依赖前端框架。
- admin 支持数据源管理、检测、预览、单源采集、采集记录、raw 数据核验和候选源过滤。
- 本地后台便于面试演示完整数据治理流程。
