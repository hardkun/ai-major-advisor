# 面试口述稿

## 30 秒版本

我做的项目叫 AI专业择校助手，是一个 AI 相关专业推荐与招生数据采集系统。它不是普通 CRUD，也不是简单聊天机器人，而是把公开招生数据采集、人工核验、规则推荐和 AI 解释串成一个闭环。用户输入分数、位次、科类和目标方向后，系统基于已核验数据返回参考推荐，AI 只负责解释学习重点、就业方向和风险提示。当前数据是部分高校公开数据验证集，不作为真实志愿填报依据。

## 1 分钟版本

这个项目的目标是做一个可部署、可演示、能体现 AI 应用开发能力的作品集项目。场景选的是 AI 相关专业择校，因为它既有数据需求，也有解释需求，而且属于高责任场景。我的设计不是让大模型直接推荐学校，而是先采集公开招生数据，进入 raw 表，再通过 admin 后台人工核验，核验后写入 admissions 正式表。推荐接口只查正式数据，根据位次和专业方向做冲稳保匹配。LLM 只负责把推荐结果解释得更自然，比如学习重点、适合人群、就业方向和风险提示。这个项目重点展示的是后端、数据管线、AI 接入和人机协同闭环。

## 3 分钟版本

### 1. 项目背景

我做这个项目，是想做一个比普通 CRUD 或聊天机器人更完整的 AI 应用作品。AI 相关专业这几年很热，但专业名称、方向标签、就业路径和招生数据都比较分散。这个场景很适合展示数据采集、推荐系统和大模型解释能力。不过我也明确把它定位为作品集项目，当前数据是部分高校公开数据验证集，不作为真实志愿填报依据。

### 2. 业务流程

用户侧流程是：输入省份、分数、位次、科类和目标 AI 方向，系统从已核验 admissions 数据中匹配院校专业，返回冲稳保、最低分、最低位次、数据来源和风险提示。如果开启 AI，系统会生成专业解释和就业方向说明。

后台流程是：先配置或发现公开数据源，然后做可采性检测和采集预览，再通过 HTML、Excel 或 PDF 解析器把数据写入 raw_admission_records。raw 数据不会直接用于推荐，必须在 admin 后台人工核验后，才写入正式 admissions 表。

### 3. 技术实现

后端用 FastAPI，按 routers、schemas、crud、services 分层。数据库用 SQLite，核心表包括 schools、majors、admissions、raw_admission_records、raw_data_sources、collector_runs 和 reports。前端有微信小程序，用于输入和展示推荐结果；还有一个原生 HTML/CSS/JS 的 admin 后台，用于数据源管理、采集预览、采集日志和人工核验。AI 部分接 OpenAI-compatible LLM API，输出结构化解释。

### 4. 难点和取舍

难点主要在数据采集。高校官网结构不统一，有普通表格、多行表头、合并单元格、宽表、Excel 附件和 PDF。所以我做了字段映射、表头自动识别、采集预览、跳过原因统计和 raw/admissions 双层数据结构。后来我也做了取舍：不继续追求全量高校数据，而是把项目收束为可演示、可解释、边界清楚的作品集。

### 5. 项目价值

这个项目展示的不是“AI 替人做决定”，而是“AI 如何安全地进入业务流程”。推荐由规则和已核验数据决定，AI 只做解释，后台保留人工核验和数据追溯。它体现了我的后端开发、数据处理、AI 接入、产品边界意识和工程闭环能力。
