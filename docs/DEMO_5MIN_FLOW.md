# 5 分钟面试演示流程

## 总目标

在 5 分钟内让面试官看到：这个项目不是聊天 Demo，而是一个包含数据采集、人工核验、推荐接口、AI 解释和后台治理的完整 AI 应用作品。

## 第 1 步：README 顶部项目定位（约 40 秒）

### 操作

打开 `README.md`，展示“30 秒看懂项目”区域。

### 要说的话

“这个项目叫 AI专业择校助手，定位是 AI 相关专业推荐与招生数据采集系统。它不是完整志愿填报系统，也不承诺录取概率。我的重点是展示公开招生数据采集、人工核验、推荐匹配和 AI 解释生成的业务闭环。”

### 想证明的能力

- 能清楚定义项目边界。
- 能把技术项目包装成可理解的业务场景。
- 对高责任场景有风险意识。

## 第 2 步：Swagger recommend 接口（约 70 秒）

### 操作

打开：

```text
http://127.0.0.1:8000/docs
```

找到 `POST /recommend`，使用示例请求：

```json
{
  "province": "四川",
  "score": 603,
  "rank": 22000,
  "subject_type": "物理类",
  "target_direction": "AI算法",
  "use_ai": true
}
```

### 要说的话

“推荐接口接收省份、分数、位次、科类和目标方向。推荐排序不是由 AI 决定，而是基于已核验 admissions 数据和规则算法。AI 只负责解释推荐结果，所以返回里会看到学校、专业、冲稳保、最低分、最低位次、数据来源、核验状态和 ai_explanation。”

### 想证明的能力

- RESTful API 设计能力。
- 推荐接口结构化输出能力。
- AI 接入边界控制能力。

## 第 3 步：admin 后台 raw/admissions 双层数据（约 80 秒）

### 操作

打开：

```text
http://127.0.0.1:8000/admin
```

展示 raw 数据核验区域，点开一条 raw 记录。

### 要说的话

“这里是后台数据治理流程。自动采集的数据不会直接进入推荐库，而是先进入 raw_admission_records，状态是 pending。管理员核验通过后才写入 admissions，recommend 只使用 admissions。这样可以避免解析错误或未核验数据影响用户结果。”

### 想证明的能力

- 数据建模能力。
- Human-in-the-loop 流程设计能力。
- 数据质量和可追溯意识。

## 第 4 步：覆盖率报告 / 采集日志（约 60 秒）

### 操作

在 admin 中展示：

- collector_runs 采集日志
- coverage_report 覆盖率报告
- gap diagnosis 缺口诊断

### 要说的话

“我还做了采集日志和覆盖率报告，用来判断哪些数据源采集成功、哪些学校已有数据、哪些还有缺口。后期我没有继续盲目追求全量高校，而是通过报告把项目收束成部分真实公开数据验证集。”

### 想证明的能力

- 数据工程进度管理能力。
- 问题诊断和项目取舍能力。
- 不盲目堆功能，有交付意识。

## 第 5 步：小程序结果页（约 60 秒）

### 操作

打开微信开发者工具中的小程序，输入示例参数，展示推荐结果页和报告页。

### 要说的话

“这是用户侧展示。用户输入分数、位次和方向后，可以看到推荐卡片、AI 解释和风险提示。报告页也做了免费版和完整报告结构，支付是 mock-pay，只用于演示产品流程。”

### 想证明的能力

- 前后端联动能力。
- 小程序产品流程实现能力。
- 能把后端能力包装成可展示的用户体验。

## 收尾话术（约 30 秒）

“所以这个项目的重点不是覆盖多少高校，而是展示一个 AI 应用从数据来源、数据治理、推荐接口到 AI 解释和前端展示的完整闭环。当前数据是验证集，我也在文档里明确说明不作为真实志愿填报依据。”
