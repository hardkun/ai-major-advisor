# STAR 面试故事

## 1. 从 0 到 1 搭建 AI 专业择校助手

### Situation 背景

我想做一个能体现 AI 应用开发能力的作品集项目，但不希望只是简单聊天机器人。

### Task 任务

目标是做出一个可运行、可演示、业务闭环完整的 AI 相关专业择校参考系统。

### Action 行动

我先搭建 FastAPI + SQLite 后端，实现 schools、majors、admissions 和基础 CRUD；再做 recommend 接口、AI 解释服务、报告结构和小程序前端；随后补充 admin 后台和数据采集核验流程。

### Result 结果

项目形成了从数据采集、人工核验、推荐匹配、AI 解释到报告展示的完整闭环，适合作为 AI 应用开发和后端工程能力展示。

### 可用于回答哪些面试问题

- 介绍一下你的项目
- 这个项目和普通 CRUD 有什么区别
- 这个项目体现了你哪些能力

## 2. 发现全量招生数据采集难度过高后及时调整项目边界

### Situation 背景

项目中期我尝试扩展更多高校招生数据源，但发现官网结构差异大，搜索结果也容易混入非目标页面。

### Task 任务

我需要判断是继续追求全量数据，还是收束项目，让它更适合作品集展示。

### Action 行动

我增加了覆盖率报告、缺口诊断和候选源官方过滤，并明确把项目定位为部分高校公开数据验证集，不再盲目追求全量覆盖。

### Result 结果

项目边界更清晰，演示更稳定，README 和 docs 中也明确说明数据不作为真实志愿填报依据。

### 可用于回答哪些面试问题

- 为什么没有继续做全量高校数据
- 项目有哪些不足
- 如果继续优化会怎么做

## 3. 设计 raw/admissions 双层数据表防止脏数据影响推荐

### Situation 背景

自动采集的数据可能存在解析错误、重复记录、字段错位或来源不可靠问题。

### Task 任务

需要保证推荐接口只使用可靠数据，同时保留自动采集效率。

### Action 行动

我设计了 raw_admission_records 和 admissions 双层表。采集数据默认进入 raw 表，状态为 pending；管理员核验通过后，才写入 admissions 正式推荐表。

### Result 结果

推荐接口只依赖正式 admissions 数据，系统既保留了自动采集能力，又降低了脏数据影响推荐结果的风险。

### 可用于回答哪些面试问题

- 为什么 raw 和 admissions 分开
- 人工核验有什么价值
- 如何保证推荐结果可追溯

## 4. 通过 admin 后台和覆盖率报告提升数据治理能力

### Situation 背景

随着数据源和 raw 记录增多，只靠 Swagger 或脚本操作效率很低，也不利于演示。

### Task 任务

需要一个本地管理后台，让数据源、采集、核验和报告都能可视化操作。

### Action 行动

我使用原生 HTML/CSS/JS 搭建 admin 后台，支持数据源管理、采集预览、单源采集、采集日志、raw 数据详情、核验和覆盖率报告。

### Result 结果

后台让项目从“接口集合”变成了可演示的数据治理系统，也更容易向面试官说明业务流程。

### 可用于回答哪些面试问题

- admin 后台的意义是什么
- 覆盖率报告有什么用
- 这个项目和普通 CRUD 有什么区别

## 5. 将机械行业工程经验迁移到 AI 应用开发项目中

### Situation 背景

我之前有机械/压力容器相关背景，转向 AI 应用开发时，需要证明工程思维可以迁移。

### Task 任务

我需要做一个能体现流程控制、风险意识和系统设计能力的 AI 项目。

### Action 行动

在项目中，我没有让 AI 直接决策，而是设计了数据来源追溯、人工核验、推荐规则、AI 解释边界和风险提示。

### Result 结果

项目体现了工程行业中重视校验、流程和安全边界的思维，也证明我能把行业问题转化为 AI 应用系统。

### 可用于回答哪些面试问题

- 机械背景如何转 AI 应用开发
- 如何控制 AI 幻觉
- 你从项目中学到了什么
